# Persistence with Docker

## The Problem: Data Disappears When Containers Stop

Remember what we learned about containers? They're isolated, temporary, and can be created and destroyed instantly. But what if your data is important? What if you're running a database or saving files?

Here's the scary truth: **When you delete a container, all its data disappears forever.**

Let's prove this, then learn how to fix it.

---

## Understanding Container Data

### Without Persistence: Data Loss

When you create a file inside a container:

```bash
docker run --name temp-test alpine sh -c "echo 'Hello from the container' > /myfile.txt"
```

The file is saved to the **container's writable layer** – the temporary storage that exists only while the container is running.

Delete the container:

```bash
docker rm temp-test
```

Try to access the file from a new container:

```bash
docker run alpine ls /myfile.txt
```

**Result**: `ls: /myfile.txt: No such file or directory`

The file is gone. The container was deleted, and its data went with it.

### The Solution: Docker Volumes

A **Docker Volume** is **permanent storage** on your computer. It's separate from the container, so it survives even when the container is destroyed.

**Think of it like this:**

- **Container** = A temporary apartment (deleted when you move out)
- **Volume** = A storage unit (stays there even after you leave)

---

## How Volumes Work: The Mount Syntax

When you attach a volume to a container, you use the `-v` flag. Let's break down the syntax:

```bash
docker run -v [VOLUME_NAME]:[CONTAINER_PATH] [IMAGE]
```

### Volume Mounting Format

| Part               | Meaning                                | Example      |
| ------------------ | -------------------------------------- | ------------ |
| **VOLUME_NAME**    | Name of your volume (or host path)     | `my-storage` |
| **:**              | Separator (always colon)               | `:`          |
| **CONTAINER_PATH** | Where inside the container it connects | `/data`      |

### Real Example

```bash
docker run -v my-storage:/data alpine
```

This means:

- **my-storage** = The volume on your hard drive
- **:/data** = Mount it to `/data` inside the container
- Now anything saved to `/data` inside the container is actually saved on your hard drive

### Another Example

```bash
docker run -v database-volume:/var/lib/mysql mysql
```

This mounts a MySQL database volume:

- **database-volume** = Volume name
- **/var/lib/mysql** = Where MySQL stores its data inside the container

Any database changes are saved to `database-volume` on your computer.

## Practical Example: Creating & Using Volumes

### Step 1: Create a Volume

First, create a volume on your computer:

```bash
docker volume create my-storage
```

This creates a storage location that Docker manages for you.

### Step 2: Run a Container with the Volume

Now run a container and attach that volume:\*\*

### Step 2: Run a Container with the Volume

Now run a container and attach that volume:

```bash
docker run -it --name persistent-test -v my-storage:/data alpine sh
```

Let's break this down:

- `-it` = Interactive mode (stay inside the terminal)
- `--name persistent-test` = Give the container a name
- `-v my-storage:/data` = Mount the volume to `/data` inside the container
- `alpine sh` = Run Alpine Linux with shell access

You're now **inside the container** shell.

### Step 3: Save Data to the Volume

Inside the container, create a file in the `/data` folder:

```bash
echo "This data will survive!" > /data/save_me.txt
```

Exit the container:

```bash
exit
```

### Step 4: Prove the Data Persists

Delete the container:

```bash
docker rm persistent-test
```

The container is gone. But the data is safe! Now create a **brand new** container and attach the **same** volume:

```bash
docker run --rm -v my-storage:/data alpine cat /data/save_me.txt
```

**Result**: It prints `This data will survive!`

Even though you deleted the original container, the file is still there because it was saved to the volume (your hard drive).

## Comparing Volume Methods

### Method 1: Named Volumes (Recommended)

```bash
docker volume create my-app-data
docker run -v my-app-data:/data my-app
```

### Method 2: Host Path Binding

You can also mount a specific folder from your computer:

```bash
docker run -v /home/user/my-data:/data my-app
```

This mounts your computer's `/home/user/my-data` folder directly into the container.

---

## Real-World Examples

### Database with Persistent Data

```bash
# Create a volume for the database
docker volume create mysql-data

# Run MySQL container with the volume
docker run -v mysql-data:/var/lib/mysql -e MYSQL_ROOT_PASSWORD=secret mysql:latest
```

Now your database data persists even if the container crashes.

### Web Application with Uploads

```bash
# Create a volume for user uploads
docker volume create app-uploads

# Run your app with the volume
docker run -v app-uploads:/app/uploads my-web-app
```

User uploads are saved to the volume and survive container deletion.

---

## Quick Comparison

| Scenario                          | Without Volume                           | With Volume                            |
| --------------------------------- | ---------------------------------------- | -------------------------------------- |
| **Create a file**                 | File saved to container's writable layer | File saved to volume (your hard drive) |
| **Stop container**                | Container stops, data stays in container | Container stops, data stays on disk    |
| **Delete container**              | **Data is deleted**                      | **Data survives** ✓                    |
| **Create new container**          | Starts empty                             | Can mount same volume and see old data |
| **Share data between containers** | Not possible                             | Easy with same volume                  |

---
**Volumes are your safety net.** Any data you want to keep should be saved to a volume. Containers are temporary, volumes are permanent.

- No volume = Data disappears with container
- With volume = Data survives forever (until you delete the volume)
