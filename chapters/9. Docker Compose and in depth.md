# Understanding Docker Compose: A Complete Tutorial

## Part 1: Conceptual Understanding

## Introduction: What Is Docker Compose, Really?

Imagine you're building a restaurant. You don't just have a kitchen—you have a kitchen, a dining area, a storage room, and a manager who coordinates everything. Docker Compose is like that manager. It takes multiple "services" (the different departments) and makes them work together seamlessly, as one unified system.

Without Docker Compose, you'd have to manually start each service (containerized part) one by one, tell them how to find each other, and manage all their connections. With Docker Compose, you describe what you want in one file, and it handles all the complexity.

#### **What Is Each Dockerfile For? (The Actual Code)**

**1. Frontend Dockerfile (in the `frontend/` folder)**

```dockerfile
FROM node:lts-buster AS development
WORKDIR /usr/src/app
COPY package.json /usr/src/app
COPY package-lock.json /usr/src/app

COPY . /usr/src/app
```

**What each line does:**

| Line                   | What It Does                                   | Why?                                                |
| ---------------------- | ---------------------------------------------- | --------------------------------------------------- |
| `FROM node:lts-buster` | Starts with Node.js pre-installed              | No need to install Node from scratch                |
| `WORKDIR /usr/src/app` | Sets the "working directory" inside container  | All commands run from this folder                   |
| `COPY package.json`    | Copies dependency list from your computer      | Container needs to know what packages to install    |
| `RUN npm ci`           | Installs all JavaScript packages               | `npm ci` is safer than `npm install` for production |
| `COPY . /usr/src/app`  | Copies your entire React application code      | Container now has the app to run                    |
| `EXPOSE 3000`          | Tells Docker this service listens on port 3000 | Documentation (not enforced, just informational)    |
| `CMD ["npm", "start"]` | The command that runs when container starts    | Starts your React dev server                        |

**The Story**: "Start with Node.js, go to the app folder, install dependencies, copy the code, expose port 3000, and run the app."

### **Breaking Down the Compose File (The Actual Code)**

Here's what the compose file looks like for this project:

```yaml
services:
  frontend:
    build:
      context: frontend
      target: development
    ports:
      - 3000:3000
    stdin_open: true
    volumes:
      - ./frontend:/usr/src/app
      - /usr/src/app/node_modules
    restart: always
    networks:
      - react-express
    depends_on:
      - backend

  backend:
    restart: always
    build:
      context: backend
      target: development
    volumes:
      - ./backend:/usr/src/app
      - /usr/src/app/node_modules
    depends_on:
      - mongo
    networks:
      - express-mongo
      - react-express
    expose:
      - 3000

  mongo:
    restart: always
    image: mongo:4.2.0
    volumes:
      - mongo_data:/data/db
    networks:
      - express-mongo
    expose:
      - 27017

networks:
  react-express:
  express-mongo:

volumes:
  mongo_data:
```

**Breaking It Down Section by Section:**

#### **The `frontend` Service**

```yaml
frontend:
  build:
    context: frontend
    target: development
```

- **`build:`** - "Build this service from a Dockerfile"
- **`context: frontend`** - "The Dockerfile is in the `./frontend` folder"
- **`target: development`** - "Use the `development` stage from the Dockerfile" (multi-stage builds)

```yaml
ports:
  - 3000:3000
```

- **First 3000** - External port (your computer)
- **Second 3000** - Internal port (inside container)
- **Translation**: When you visit `localhost:3000`, it connects to the container's port 3000

```yaml
volumes:
  - ./frontend:/usr/src/app
  - /usr/src/app/node_modules
```

- **First volume** - Bind mount your frontend code folder to the container
  - Your code changes instantly appear in the container
  - Great for development (hot reload)
- **Second volume** - Keep node_modules in container (don't bind mount it)
  - Why? Windows path differences cause issues
  - Let Docker manage it instead

```yaml
restart: always
```

- If the container crashes, restart it immediately
- `always` means "restart no matter what"
- Other options: `on-failure`, `unless-stopped`, `no`

```yaml
networks:
  - react-express
```

- Add this service to the `react-express` network
- Allows communication with frontend and backend

```yaml
depends_on:
  - backend
```

- Don't start frontend until backend is ready
- Prevents "can't connect to backend" errors

---

#### **The `backend` Service**

```yaml
backend:
  restart: always
  build:
    context: backend
    target: development
```

- Same concept as frontend, but builds from the `./backend` folder

```yaml
volumes:
  - ./backend:/usr/src/app
  - /usr/src/app/node_modules
```

- Backend code is bind-mounted for live development
- Node modules stay isolated in the container

```yaml
depends_on:
  - mongo
```

- Backend waits for MongoDB to be ready
- Without this, backend might crash trying to connect to a non-existent database

```yaml
networks:
  - express-mongo
  - react-express
```

- Backend is on TWO networks:
  - `express-mongo` to talk to MongoDB
  - `react-express` to talk to frontend
- Backend is the "middle man" between frontend and database

```yaml
expose:
  - 3000
```

- "This service listens on port 3000"
- `expose` is different from `ports`:
  - `ports` = Accessible from outside Docker (your computer)
  - `expose` = Accessible only to other services (internal only)

---

#### **The `mongo` Service**

```yaml
mongo:
  restart: always
  image: mongo:4.2.0
```

- `image:` instead of `build:` - Use pre-built image instead of custom Dockerfile
- `mongo:4.2.0` - MongoDB version 4.2.0 from Docker Hub

```yaml
volumes:
  - mongo_data:/data/db
```

- MongoDB stores data in `/data/db` folder inside container
- This is mapped to a Docker named volume called `mongo_data`
- Data persists even if container is deleted (you can recreate it and data is still there)

```yaml
networks:
  - express-mongo
```

- Only on the `express-mongo` network
- Frontend cannot directly access database (extra security)

```yaml
expose:
  - 27017
```

- Exposes port 27017 (MongoDB's default port)
- Only internal services can access it

---

#### **The `networks` Section**

```yaml
networks:
  react-express:
  express-mongo:
```

- Creates two isolated networks
- Each service is assigned to specific networks based on what it needs to talk to
- Frontend can only reach backend (react-express network)
- Database can only be reached by backend (express-mongo network)
- **Security benefit**: Frontend code can't directly query database even if hacked

---

#### **The `volumes` Section**

```yaml
volumes:
  mongo_data:
```

- Declares a named volume called `mongo_data`
- This volume persists data permanently
- When container stops, data stays
- When container restarts, data is there

---

### **How Docker Compose Orchestrates Everything**

When you run Docker Compose with this project, here's the beautiful magic that happens:

#### **Step 1: Building the Services**

Docker Compose reads the compose file and says:

- "Build the frontend from the Dockerfile in the `frontend/` folder"
- "Build the backend from the Dockerfile in the `backend/` folder"
- "Download the MongoDB image from Docker Hub and prepare it"

#### **Step 2: Creating Networks**

Docker Compose creates **two separate networks**:

- **react-express network**: Connects frontend ↔ backend
- **express-mongo network**: Connects backend ↔ database

Why two networks? **Security through isolation**. The frontend never directly talks to the database—it always goes through the backend. This is like having a bouncer at a restaurant who checks credentials.

#### **Step 3: Starting Services in Order (The Dependency Chain)**

This is the **critical** part. Docker Compose reads the `depends_on` declarations and works **backwards** to determine startup order.

**The Compose File Says:**

```yaml
frontend:
  depends_on:
    - backend # Frontend says: "I need backend first"

backend:
  depends_on:
    - mongo # Backend says: "I need mongo first"

mongo:
  # No depends_on - I don't need anything!
```

**Docker Compose figures out:**

- Mongo has NO dependencies → **Start mongo first** ✅
- Backend depends on mongo → **Start backend second** ✅
- Frontend depends on backend → **Start frontend third** ✅

**Visual Startup Order:**

```
Startup Timeline
────────────────────────────────────────────────→

1. MongoDB starts
   (no dependencies, goes first)
   ↓
2. Backend starts
   (waits for mongo to be ready)
   ↓
3. Frontend starts
   (waits for backend to be ready)

Final Result:
  Frontend ←→ Backend ←→ MongoDB
  (User Interface)  (API Logic)  (Data Storage)
```

**Why This Order Matters:**

| Order              | Why                                                                                                          |
| ------------------ | ------------------------------------------------------------------------------------------------------------ |
| **Database First** | Backend needs a working database to start properly. If mongo isn't ready, backend crashes trying to connect. |
| **Backend Second** | Frontend needs a working API to call. If backend isn't ready, frontend crashes trying to fetch data.         |
| **Frontend Third** | It's the entry point. By the time it starts, the whole backend infrastructure is ready.                      |

**The Opposite Would Be Disaster:**

If we did it backwards (Frontend → Backend → Database):

1. Frontend starts, tries to call `backend:3000`... backend doesn't exist yet! ❌ CRASH
2. Backend starts, tries to connect to `mongo:27017`... mongo doesn't exist yet! ❌ CRASH
3. Database finally starts, but everything else already crashed ❌ CHAOS

**Restaurant Analogy:**

```
Wrong Order:
Open dining room → Kitchen not ready → Customers angry
Open kitchen → Pantry not stocked → Food can't be made → Disaster
Finally stock pantry → Too late, everything collapsed

Right Order:
Stock pantry first → Kitchen can operate → Dining room serves customers
                                        ↓
                                      SUCCESS
```

---

#### **Step 4: Service Discovery (The Magic Part)**

Here's where it gets really cool. Inside each container, services don't need IP addresses—they use **service names**:

- The frontend talks to `backend` (not `192.168.1.50`)
- The backend talks to `mongo` (not `192.168.1.51`)

Docker Compose runs an internal DNS server that translates these names to the actual containers. It's like a phone book for containers.

#### **Step 5: Volume Mounting**

- **Frontend folder** is linked to the container, so changes in your code are instantly reflected
- **Backend folder** is linked similarly for live development
- **mongo_data volume** is persistent storage for the database (data survives even if containers stop)

---

### **What Happens When You Run It**

When you execute `docker compose up`:

1. Three isolated containers spin up
2. They're connected via networks so they can "talk"
3. Frontend is accessible at `localhost:3000`
4. Frontend requests go through the network to the backend
5. Backend queries are made to MongoDB through the other network
6. All data is persisted in the mongo_data volume
7. Logs from all three services stream to your terminal
8. Everything runs in harmony, managed by one simple file

## Project 2: Nginx-NodeJS-Redis

### Load Balancing and Caching Made Simple

#### **What Is This Project?**

This project shows a real-world scaling scenario:

- **Multiple Backend Replicas**: Two identical Node.js web servers (web1 and web2)
- **Load Balancer**: Nginx that distributes traffic between them
- **Cache Layer**: Redis to speed up responses

Think of it like a pizza restaurant with two kitchens:

- When orders come in, a manager (Nginx) sends each order to whichever kitchen is less busy
- Both kitchens have a spice rack (Redis) they can quickly grab from instead of making everything from scratch

---

### **Folder Structure Overview**

```
nginx-nodejs-redis/
├── web/               # The Node.js web application code
│   └── (Contains app code and a Dockerfile for the web servers)
├── nginx/             # Nginx configuration
│   └── (Contains Dockerfile to set up the reverse proxy)
├── compose.yaml       # The orchestration blueprint
└── README.md         # Instructions
```

#### **What Is Each Dockerfile For?**

**1. Web Dockerfile (in the `web/` folder)**

```dockerfile
FROM node:14.17.3-alpine3.14
WORKDIR /usr/src/app
COPY package.json package-lock.json ./
RUN npm ci
COPY ./server.js ./
CMD ["npm","start"]
```

**What each line does:**

| Line                                     | What It Does                                        |
| ---------------------------------------- | --------------------------------------------------- |
| `FROM node:14.17.3-alpine3.14`           | Start with Node.js v14 in Alpine Linux (tiny image) |
| `WORKDIR /usr/src/app`                   | Set working directory                               |
| `COPY package.json package-lock.json ./` | Copy dependency files                               |
| `RUN npm ci`                             | Install packages                                    |
| `COPY ./server.js ./`                    | Copy the web server code                            |
| `CMD ["npm","start"]`                    | Run the server when container starts                |

**Key point**: This same Dockerfile is used to build **both web1 and web2**. They're identical in code, but separate running containers.

---

**2. Nginx Dockerfile (in the `nginx/` folder)**

```dockerfile

COPY conf /etc/nginx/conf.d/default.conf
```

**What it does:**

- Starts with Nginx pre-installed
- Copies your custom configuration that tells Nginx to load-balance between web1 and web2
- The config file knows about `web1` and `web2` by service name (service discovery!)

**Why so simple?** Nginx configuration is just a text file. No building logic needed.

---

**3. Redis Service**

- **No custom Dockerfile needed**: Uses official `redislabs/redismod` image
- This is a pre-made Redis with extra modules
- Ready to go, no configuration needed

---

### **Breaking Down the Compose File**

```yaml
services:
  redis:
    image: "redislabs/redismod"
    ports:
      - "6379:6379"

  web1:
    restart: on-failure
    build: ./web
    hostname: web1
    ports:
      - "81:5000"

  web2:
    restart: on-failure
    build: ./web
    hostname: web2
    ports:
      - "82:5000"

  nginx:
    build: ./nginx
    ports:
      - "80:80"
    depends_on:
      - web1
      - web2
```

**Breaking It Down:**

#### **The `redis` Service**

```yaml
redis:
  image: "redislabs/redismod"
  ports:
    - "6379:6379"
```

- `image:` instead of `build:` - Uses a pre-made image
- Exposes port 6379 (Redis default) to your computer
- All services can access it by the name `redis`

---

#### **The `web1` Service**

```yaml
web1:
  restart: on-failure
  build: ./web
  hostname: web1
  ports:
    - "81:5000"
```

- **`restart: on-failure`** - Only restart if it crashes (not always)
- **`build: ./web`** - Build from the Dockerfile in `./web` folder
- **`hostname: web1`** - Give this container a hostname/name
- **`ports: - '81:5000'`** - Map your port 81 to container's port 5000
  - This lets you access it directly at `localhost:81`

---

#### **The `web2` Service**

```yaml
web2:
  restart: on-failure
  build: ./web
  hostname: web2
  ports:
    - "82:5000"
```

**Identical to web1**, except:

- Different hostname (`web2` instead of `web1`)
- Different external port (`82` instead of `81`)
- Same Dockerfile, different container instance

---

#### **The `nginx` Service**

```yaml
nginx:
  build: ./nginx
  ports:
    - "80:80"
  depends_on:
    - web1
    - web2
```

- **`build: ./nginx`** - Build from Dockerfile in `./nginx` folder
- **`ports: - '80:80'`** - Expose port 80 (standard web port)
- **`depends_on: [web1, web2]`** - Start nginx AFTER web1 and web2 are ready

**The magic**: Nginx's config file references `web1` and `web2` by hostname. Docker DNS automatically routes `web1` to the web1 container and `web2` to the web2 container.

---

### **How It All Works Together**

1. Redis starts (the cache)
2. web1 starts with hostname `web1` on internal port 5000
3. web2 starts with hostname `web2` on internal port 5000
4. Nginx starts and loads its config, which knows about both web servers
5. User visits `localhost:80`
6. Nginx receives the request and decides: send to web1 or web2?
7. Nginx forwards to the selected web server
8. Web server processes request and stores result in Redis
9. Response goes back through Nginx to user
10. Next user might get cached result from Redis (no web server needed!)

---

#### **Step 1: Creating Multiple Instances**

This is a key feature. Docker Compose doesn't just start one web server—it starts **two identical copies**:

- **web1**: Node.js app running on internal port 5000, exposed as external port 81
- **web2**: Same Node.js app running on internal port 5000, exposed as external port 82

Both are built from the same Dockerfile but run as separate containers. It's like baking the same cake recipe twice—two cakes, two flavors of the same thing.

#### **Step 2: Redis Cache Starts**

Redis launches and waits for the web servers to store data in it.

#### **Step 3: Nginx Load Balancer Starts**

Nginx starts with knowledge of both web services:

- It discovers `web1` and `web2` by name
- It's configured to route incoming traffic to both
- Users only see one entry point: `localhost:80`

#### **Step 4: Smart Traffic Distribution**

When a user visits the website:

1. Their request hits Nginx on port 80
2. Nginx checks if it's a cache hit in Redis (fast!)
3. If not cached, Nginx picks one of the web servers (web1 or web2)
4. The web server processes the request
5. Results are cached in Redis
6. Response goes back through Nginx to the user

#### **Step 5: Failure Tolerance**

If web1 crashes:

- Nginx detects this and stops sending traffic to it
- All requests go to web2
- The system keeps running!
- You can restart web1 without affecting users

---

### **What Happens When You Run It**

When you execute `docker compose up`:

1. Redis container starts (the cache)
2. web1 container starts (Node.js app #1)
3. web2 container starts (Node.js app #2)
4. Nginx container starts (the load balancer)
5. Nginx knows about both web servers through service discovery
6. You visit `localhost:80` and get served by either web1 or web2
7. Subsequent visits might be served from cache without hitting the web servers
8. If one server struggles, the other picks up the slack
9. All coordinated by one compose file—no manual management needed

## Project 3: Nginx-Flask-MySQL

### Enterprise-Grade Multi-Tier Application

#### **What Is This Project?**

This is the most production-like example, showing:

- Health checks (ensure services are actually healthy)

Think of it like a bank:

- **Nginx** is the security checkpoint and greeter
- **Flask** is the teller and account manager
- **MySQL** is the vault holding customer money

### **Breaking Down the Compose File**

```yaml
services:
  db:
    image: mariadb:10-focal
    command: "--default-authentication-plugin=mysql_native_password"
    restart: always
    healthcheck:
      test:
        [
          "CMD-SHELL",
          'mysqladmin ping -h 127.0.0.1 --password="$$(cat /run/secrets/db-password)" --silent',
        ]
      interval: 3s
      retries: 5
      start_period: 30s
    secrets:
      - db-password
    volumes:
      - db-data:/var/lib/mysql
    networks:
      - backnet
    environment:
      - MYSQL_DATABASE=example
      - MYSQL_ROOT_PASSWORD_FILE=/run/secrets/db-password
    expose:
      - 3306
      - 33060

  backend:
    build:
      context: backend
      target: builder
    restart: always
    secrets:
      - db-password
    ports:
      - 8000:8000
    networks:
      - backnet
      - frontnet
    depends_on:
      db:
        condition: service_healthy

  proxy:
    build: proxy
    restart: always
    ports:
      - 80:80
    depends_on:
      - backend
    networks:
      - frontnet

volumes:
  db-data:

secrets:
  db-password:
    file: db/password.txt

networks:
  backnet:
  frontnet:
```

---

#### **The `db` Service (Most Complex)**

```yaml
db:
  image: mariadb:10-focal
  command: "--default-authentication-plugin=mysql_native_password"
```

- Uses MariaDB (MySQL-compatible) version 10
- Passes a custom command to change the auth plugin

```yaml
healthcheck:
  test:
    [
      "CMD-SHELL",
      'mysqladmin ping -h 127.0.0.1 --password="$$(cat /run/secrets/db-password)" --silent',
    ]
  interval: 3s
  retries: 5
  start_period: 30s
```

**This is CRITICAL:**

- **`test:`** - Runs a command to check if MySQL is healthy
- **`mysqladmin ping`** - Tries to connect to MySQL
- **`interval: 3s`** - Check every 3 seconds
- **`retries: 5`** - Give it up to 5 chances
- **`start_period: 30s`** - Wait 30 seconds before starting checks
- **Purpose**: Backend WON'T start until this passes!
- **Why?** Prevents "can't connect to database" crashes

```yaml
secrets:
  - db-password
```

- Access a secret called `db-password` (defined at bottom of file)
- Secrets are passed as files, not environment variables
- Much more secure than hardcoding passwords

```yaml
environment:
  - MYSQL_DATABASE=example
  - MYSQL_ROOT_PASSWORD_FILE=/run/secrets/db-password
```

- Create a database called `example` automatically

- MySQL reads the password from the file, not the environment

```yaml
volumes:
  - db-data:/var/lib/mysql
```

- Store MySQL data in the `db-data` named volume
- Data persists even if container is deleted

```yaml
networks:
  - backnet
```

```yaml
backend:
```

### **Folder Structure Overview**

**1. Backend Dockerfile (in the `backend/` folder)**

- **Job**: Packages the Flask Python application

  - Prepares the application to receive API requests
- **Think of it as**: Setting up a teller booth with training and equipment

**2. Proxy Dockerfile (in the `proxy/` folder)**

- **Job**: Configures Nginx as the entry point and reverse proxy
- **What it does**:
  - Loads Nginx
  - Applies a custom configuration that routes traffic to the Flask backend
  - Handles all incoming HTTP requests
  - Forwards them to the correct backend service
- **Think of it as**: Training a security guard on protocols and routing

**3. Database Service (MySQL)**

- **No custom Dockerfile needed**: Uses official MySQL/MariaDB image
- **Job**: Stores all application data persistently
- **Special feature**: Has **health checks** to ensure it's actually ready
- **Think of it as**: A pre-trained vault manager

### **How Docker Compose Orchestrates Everything**

#### **Step 1: Creating Networks for Security**

This project creates **two networks**:

- **backnet**: Connects backend and database (isolated from outside)
- **frontnet**: Connects proxy and backend

The proxy is the only service exposed to the internet. Everything else is hidden behind it. **Defense in depth**—a security best practice.

#### **Step 2: Health Checks (The Safeguard)**

Before starting the next service, Docker Compose checks:

- "Is MySQL actually ready to accept connections?"
  - It runs a health check command every 3 seconds
  - Waits up to 30 seconds for success
  - Won't proceed until MySQL says "I'm healthy!"

**Why this matters**: Starting Flask when MySQL isn't ready = crash. Health checks prevent this.

#### **Step 3: Secrets Management**

This project demonstrates **secure password handling**:

- Database passwords aren't hardcoded in the compose file
- They're stored separately as "secrets"
- Services receive secrets as files in their container
- Passwords are never exposed in logs or configurations
- **Think of it as**: Using a safe combination instead of writing the password on a sticky note

#### **Step 4: Service Dependencies**

The compose file explicitly states:

- Flask backend **depends_on** MySQL database
- Nginx proxy **depends_on** Flask backend

Docker Compose respects these dependencies and starts services in the correct order.

#### **Step 5: Data Persistence**

- **db-data volume**: MySQL data persists even if containers are destroyed and recreated
- Your database isn't lost if you update containers
- **Think of it as**: The vault doesn't lose money when tellers go home

#### **Step 6: Networking Magic**

- **Flask backend** connects to `db` (service name), not an IP address
- **Nginx** connects to `backend` (service name)
- Docker Compose's internal DNS translates these names to the actual containers
- Services are discoverable by name, not brittle IP addresses

### **What Happens When You Run It**

When you execute `docker compose up`:

1. Docker creates the networks (backnet and frontnet)
2. MySQL starts and begins its health check
3. Docker waits for the health check to pass (confirms DB is ready)
4. Flask backend starts and connects to MySQL using the `db` service name
5. Nginx proxy starts and is configured to forward traffic to `backend`
6. Nginx listens on port 80 (the only exposed port)
7. A user visits `localhost:80`
8. Nginx receives the request and forwards it to the Flask backend
9. Flask processes it, queries MySQL if needed
10. Response travels back through Nginx to the user
11. All logs appear in your terminal
12. Everything is secure, ordered, and coordinated

## The Magic Under the Hood: How Services Talk to Each Other

### **Service Discovery (No Hard-Coded IPs)**

Without Docker Compose, you'd have to do something like:

- "Flask, connect to database at 192.168.1.100:3306"
- "Nginx, send requests to 192.168.1.101:8000"

These IP addresses are fragile and can change.

**With Docker Compose**:

- "Flask, connect to `db`"
- "Nginx, send requests to `backend`"

Docker Compose runs an internal DNS service that translates service names to the actual containers. When a container says "I need to talk to `db`", Docker knows exactly which container that is and routes it there.

**Why this is brilliant**: Services don't need to know where other services are. They just use names. Services are truly interchangeable and portable.

---

### **Networks: Invisible Paths Between Services**

Think of Docker networks like postal routes:

- Each service is in a building
- Networks are the roads between buildings
- Services in the same network can "mail letters" to each other using service names
- Services in different networks can't directly communicate (unless explicitly connected)

In the nginx-flask-mysql example:

- Nginx is on the `frontnet` "street" and the `backnet` "street" (it's at the intersection)
- Flask is on the `backnet` "street" and the `frontnet` "street"
- MySQL is only on the `backnet` "street"
- External users can only reach Nginx (which is exposed)
- Nobody from outside can directly reach Flask or MySQL

---

### **Volumes: Memory That Survives**

Services are temporary—containers can be deleted and recreated. But data is permanent.

**Named Volumes** (like `mongo_data` or `db-data`):

- Data stored here persists even if containers are destroyed
- When you recreate containers, the data is still there
- Multiple containers can share the same volume

**Bind Mounts** (like `./frontend:/usr/src/app`):

- Your host computer's folder is directly linked to the container's folder
- Changes on your computer show up instantly in the container
- Perfect for development (change code, container picks it up immediately)

---

### **Dependencies & Startup Order**

Docker Compose respects declared dependencies. If service A `depends_on` service B:

1. Service B starts first
2. Docker Compose waits for service B to be "healthy" (if health checks are defined)
3. Only then does service A start

This prevents the chaos of services crashing because required dependencies aren't ready.

---

---

## The Full Picture: How Everything Works Together

### **Visual Flow for nginx-flask-mysql**

```
User's Browser
      ↓
   Nginx (Proxy)  ← Exposed to the internet
      ↓
   Flask (Backend) ← Hidden, only Nginx can reach
      ↓
   MySQL (Database) ← Completely hidden, only Flask can reach
```

**The Beauty**:

1. External attacks can only hit Nginx
2. Nginx is configured to only talk to Flask
3. Flask is hardened and only talks to MySQL
4. **Security through layers**

### **A Day in the Life of a User Request**

1. **Request arrives**: `curl localhost:80/api/users`
2. **Nginx receives it**: "A request for `/api/users` just came in"
3. **Nginx routes it**: "This goes to the `backend` service"
4. **Service discovery kicks in**: "The `backend` service is running on 172.18.0.2:8000"
5. **Flask receives it**: "GET /api/users, I need to query the database"
6. **Flask connects to database**: "Give me all users from `db` database"
7. **Service discovery again**: "The `db` service (MySQL) is running on 172.18.0.3:3306"
8. **MySQL executes**: "SELECT \* FROM users"
9. **MySQL responds**: Sends the user list to Flask
10. **Flask processes**: Formats the response
11. **Flask responds to Nginx**: "Here's the JSON data"
12. **Nginx responds to browser**: User sees the list of users

All of this happens in milliseconds, all coordinated by Docker Compose without you lifting a finger.

# Part 2: How to Write a Docker Compose File From Scratch

Now that you understand how compose files work, let's build one from the ground up.

## Step 1: Plan Your Services

Before writing any code, answer these questions:

1. **What services do I need?**
   - Web application? Database? Cache? Reverse proxy?
2. **Which ones need custom Dockerfiles, and which use pre-built images?**
   - If you have custom code: write a Dockerfile, use `build:` in compose
   - If using standard software: pull from Docker Hub, use `image:` in compose
3. **What ports do they use?**
   - Internal ports (inside containers)
   - External ports (on your computer)
4. **Which services need to talk to each other?**
   - This determines your networks
5. **What data needs to persist?**
   - This determines your volumes

**Example Plan for a Simple Blog:**

- Service 1: Node.js backend (custom code) → needs Dockerfile
- Service 2: PostgreSQL database (standard software) → use image
- Service 3: Nginx proxy (standard software) → use image
- Backend needs to talk to PostgreSQL
- Nginx needs to talk to Backend
- PostgreSQL data must persist
- External users only hit Nginx (port 80)

---

## Step 2: Start with the Skeleton

Every compose file has this structure:

```yaml
version: "3.8"

services:
  # Your services go here

volumes:
  # Named volumes go here

networks:
  # Custom networks go here

secrets:
  # Sensitive data goes here (optional)
```

**What each part means:**

| Section          | Purpose                                       |
| ---------------- | --------------------------------------------- |
| `version: '3.8'` | Compose file format version (3.8 is standard) |
| `services:`      | All your containers                           |
| `volumes:`       | Named volumes for persistent data             |
| `networks:`      | Custom networks for service communication     |
| `secrets:`       | Secure passwords/keys (optional)              |

---

## Step 3: Define Your Services

For each service, you need to decide: **Build or Image?**

### **Option A: Using a Pre-Built Image (Simplest)**

```yaml
services:
  database:
    image: postgres:13-alpine
    ports:
      - 5432:5432
    environment:
      POSTGRES_PASSWORD: mysecretpassword
    volumes:
      - db_data:/var/lib/postgresql/data
    networks:
      - app-network
```

**What to provide:**

- `image:` - The image name and version (from Docker Hub)
- `ports:` - External mapping (or omit if internal only)
- `environment:` - Configuration variables
- `volumes:` - Where data is stored
- `networks:` - Which networks it's on

---

### **Option B: Building from a Custom Dockerfile (Your Code)**

```yaml
services:
  backend:
    build:
      context: ./backend
      dockerfile: Dockerfile
    ports:
      - 8000:8000
    environment:
      DATABASE_URL: postgres://database:5432/mydb
    volumes:
      - ./backend:/app
    networks:
      - app-network
    depends_on:
      - database
```

**What to provide:**

- `build:` instead of `image:`
- `context:` - Folder containing the Dockerfile
- `dockerfile:` - Name of the Dockerfile (usually just "Dockerfile")
- `ports:`, `environment:`, `volumes:`, `networks:` - Same as above
- `depends_on:` - Other services this needs

---

## Step 4: Complete Example - Blog Application

Let's write a complete compose file for a simple blog:

```yaml
version: "3.8"

services:
  # The database
  postgres:
    image: postgres:13-alpine
    restart: always
    environment:
      POSTGRES_USER: blogger
      POSTGRES_PASSWORD: secretpassword
      POSTGRES_DB: blog_db
    volumes:
      - postgres_data:/var/lib/postgresql/data
    networks:
      - backend-network
    expose:
      - 5432

  # The web server (Node.js backend)
  backend:
    build:
      context: ./backend
      dockerfile: Dockerfile
    restart: always
    environment:
      DATABASE_URL: postgresql://blogger:secretpassword@postgres:5432/blog_db
      NODE_ENV: production
      PORT: 3000
    ports:
      - 3000:3000
    volumes:
      - ./backend:/app
      - /app/node_modules
    networks:
      - backend-network
      - frontend-network
    depends_on:
      - postgres

  # The reverse proxy (Nginx)
  nginx:
    build:
      context: ./nginx
      dockerfile: Dockerfile
    restart: always
    ports:
      - 80:80
      - 443:443
    networks:
      - frontend-network
    depends_on:
      - backend

volumes:
  postgres_data:

networks:
  backend-network:
  frontend-network:
```

**Let's break down what's happening:**

| Service    | Purpose    | Key Points                                                       |
| ---------- | ---------- | ---------------------------------------------------------------- |
| `postgres` | Database   | Pre-built image, data persists in volume, internal only          |
| `backend`  | API server | Custom Dockerfile, environmental variables passed, on 2 networks |
| `nginx`    | Web proxy  | Custom Dockerfile, only service exposed to internet              |

**The Flow:**

1. Postgres starts
2. Backend starts (can find `postgres` by name)
3. Backend connects to database at `postgresql://postgres:5432/...`
4. Nginx starts (knows backend is at `backend:3000`)
5. User visits `localhost:80`
6. Nginx routes to `backend:3000`
7. Backend queries PostgreSQL
8. Response returns to user

## Step 5: Common Configurations Explained

### **Restart Policies**

```yaml
restart: always                  # Restart no matter what
restart: on-failure             # Restart only if it crashes
restart: unless-stopped         # Restart unless explicitly stopped
restart: no                     # Don't restart
```

**When to use which:**

- `always` - For databases, caches, critical services
- `on-failure` - For web servers and APIs
- `no` - For one-time jobs or services you manage manually

---

### **Port Mapping**

```yaml
ports:
  - "8000:3000" # External:Internal (accessible from your computer)

expose:
  - 3000 # Only internal (other services can access)
```

**Rule of thumb:**

- Use `ports:` only for services users directly access
- Use `expose:` for internal services (database, backend APIs)
- Nginx/proxies use `ports:`, services behind them use `expose:`

---

### **Environment Variables**

```yaml
environment:
  DATABASE_URL: postgresql://postgres:5432/db
  DEBUG: "true"
  API_KEY: sk_live_xyz123
```

**Good practices:**

- Use for configuration that changes per environment
- DO NOT hardcode secrets here (use `secrets:` instead)
- Service names can be used as hostnames (e.g., `postgres` = service address)

---

### **Volumes (Persistent Data)**

```yaml
volumes:
  # Named volume (data persists, managed by Docker)
  - postgres_data:/var/lib/postgresql/data

  # Bind mount (your folder linked to container)
  - ./backend:/app

  # Anonymous volume (temporary storage in container)
  - /app/node_modules
```

**When to use each:**

| Type             | Use Case                       | Example                         |
| ---------------- | ------------------------------ | ------------------------------- |
| Named volume     | Persistent, critical data      | Database files, uploads         |
| Bind mount       | Development (live code reload) | Source code folders             |
| Anonymous volume | Temporary, isolated data       | `node_modules` (Windows issues) |

---

### **Networks (Service Communication)**

```yaml
networks:
  service-network:

# In a service:
networks:
  - service-network  # Join this network
```

**Network isolation pattern:**

```yaml
networks:
  internet-facing: # Frontend, proxy (can be reached from outside)
  internal-only: # Backend, database (hidden from outside)

services:
  nginx:
    networks:
      - internet-facing
  backend:
    networks:
      - internet-facing # Receives from nginx
      - internal-only # Talks to database
  database:
    networks:
      - internal-only # Hidden, only backend can access
```

**Why?** The backend is the gatekeeper. Frontend can't reach database even if hacked.

---

### **Dependencies and Health Checks**

#### **Understanding `depends_on` (Startup Order)**

`depends_on` controls **the order in which services start**. When you write:

```yaml
backend:
  depends_on:
    - postgres
```

You're saying: "**Don't start backend until postgres is ready.**"

**How Startup Order Works (Work Backwards):**

Look at your services and identify which have **no dependencies**:

```yaml
# Example 1: Three services
services:
  database:
    image: postgres
    # ← NO depends_on = starts FIRST

  backend:
    build: ./app
    depends_on:
      - database # ← Waits for database

  frontend:
    build: ./web
    depends_on:
      - backend # ← Waits for backend
```

**Docker reads this and determines:**

1. **database** has no dependencies → Start FIRST
2. **backend** depends on database → Start SECOND
3. **frontend** depends on backend → Start THIRD

**Visual Timeline:**

```
Time →

[database starts]
       ↓ (database ready)
       [backend starts]
              ↓ (backend ready)
              [frontend starts]
                      ↓
                      All Running!
```

**Example 2: Multiple dependencies**

```yaml
services:
  postgres:
    image: postgres
    # No depends_on

  redis:
    image: redis
    # No depends_on

  backend:
    depends_on:
      - postgres
      - redis
    # Waits for BOTH postgres AND redis
```

**Docker starts order:**

1. postgres AND redis start together (both have no dependencies)
2. backend waits for both to be ready
3. backend starts

---

#### **The Difference: `depends_on` vs `service_healthy`**

```yaml
# Simple depends_on - not recommended
depends_on:
  - postgres              # Wait for container to START

# Better - wait for service to be READY
depends_on:
  postgres:
    condition: service_healthy  # Wait until health check PASSES
```

**The problem with simple depends_on:**

- Container starts ≠ Service is ready
- Example: PostgreSQL container starts, but database is still initializing
- Your app tries to connect → CRASH ❌

**Why `service_healthy` is better:**

- Waits until the service **proves** it's working
- PostgreSQL reports: "I'm ready!"
- Your app connects safely → SUCCESS ✅

**Health check example:**

```yaml
postgres:
  image: postgres
  healthcheck:
    test: ["CMD-SHELL", "pg_isready -U postgres"] # "Are you ready?"
    interval: 5s # Check every 5 seconds
    timeout: 5s # Wait max 5 seconds for response
    retries: 5 # Try 5 times before giving up
    start_period: 10s # Give 10 seconds before first check

backend:
  depends_on:
    postgres:
      condition: service_healthy # Wait for this check to pass!
```

**Health Check Explained:**

| Setting         | Meaning                                                 |
| --------------- | ------------------------------------------------------- |
| `test:`         | Command to run (exit 0 = healthy, non-zero = unhealthy) |
| `interval:`     | How often to run the test                               |
| `timeout:`      | How long to wait for response                           |
| `retries:`      | How many failures before marking unhealthy              |
| `start_period:` | Time to wait before first check                         |

**Real-World Timeline:**

```
PostgreSQL Container Starts
    ↓
1. Wait 10 seconds (start_period)
    ↓
2. Every 5 seconds (interval), test: pg_isready
    ├─ Attempt 1: Not ready yet... ❌ (retries: 4 left)
    ├─ Attempt 2: Not ready yet... ❌ (retries: 3 left)
    ├─ Attempt 3: Not ready yet... ❌ (retries: 2 left)
    ├─ Attempt 4: Not ready yet... ❌ (retries: 1 left)
    ├─ Attempt 5: READY! ✅
    ↓
Backend Can Now Start (condition: service_healthy = true)
```

## Step 6: Common Mistakes to Avoid

### **Mistake 1: Hard-Coding the Server Address**

❌ **Wrong:**

```yaml
environment:
  DATABASE_URL: postgresql://192.168.1.50:5432/db
```

✅ **Right:**

```yaml
environment:
  DATABASE_URL: postgresql://postgres:5432/db
```

**Why?** Docker handles DNS internally. Use service names, not IPs.

---

### **Mistake 2: Exposing Everything**

❌ **Wrong:**

```yaml
database:
  ports:
    - 5432:5432 # Now anyone can try to hack your database!
```

✅ **Right:**

```yaml
database:
  expose:
    - 5432 # Only internal services can access
```

**Why?** Use `expose:` for internal services. Only expose what users need.

---

### **Mistake 3: Forgetting Volumes for Databases**

❌ **Wrong:**

```yaml
postgres:
  image: postgres:13
  # No volume! Data is lost when container stops
```

✅ **Right:**

```yaml
postgres:
  image: postgres:13
  volumes:
    - postgres_data:/var/lib/postgresql/data

volumes:
  postgres_data:
```

**Why?** Without volumes, restarting the container loses all data!

---

### **Mistake 4: Not Using Dependencies**

❌ **Wrong:**

```yaml
backend:
  environment:
    DATABASE_URL: postgresql://postgres:5432/db
  # No depends_on! Might start before postgres is ready
```

✅ **Right:**

```yaml
backend:
  environment:
    DATABASE_URL: postgresql://postgres:5432/db
  depends_on:
    postgres:
      condition: service_healthy
```

**Why?** Backend needs database to be ready, not just started.

---

### **Mistake 5: Hardcoding Secrets**

❌ **Wrong:**

```yaml
environment:
  DB_PASSWORD: mysecretpassword123
```

✅ **Right:**

```yaml
secrets:
  - db_password

environment:
  DB_PASSWORD_FILE: /run/secrets/db_password

# In secrets section:
secrets:
  db_password:
    file: ./.env.secret
```

**Why?** Secrets in `environment` show up in logs and `docker inspect` output!

```yaml
      test: ["CMD", "curl", "-f", "http://localhost:3000"]
      interval: 10s

    # Data
    volumes:
      - my_volume:/data # Named volume
      - ./src:/app/src # Bind mount

    # Configuration
    environment:
      VAR_NAME: value
      DATABASE_URL: postgresql://postgres:5432/db

    secrets:
      - my_secret

volumes:
  my_volume:

networks:
  my-network:

secrets:
  my_secret:
    file: ./.env.secret
```

## In Human Terms

Docker Compose is like a **restaurant manager**:

- You tell the manager what departments you need (kitchen, dining room, pantry)
- The manager hires and trains staff for each department
- The manager makes sure departments can communicate with each other
- The manager ensures the pantry is stocked before the kitchen starts cooking
- The manager keeps everything running smoothly
- If a department fails, the manager fixes it
- New customers come in, and the manager directs them to the right place

You don't manually manage each department—the manager does. That's Docker Compose.
